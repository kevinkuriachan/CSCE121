Approach

A Set ADT

Sets have several operations, most of which we're quite familiar with from basic mathematics:

  • union, takes two sets and produces a new third one as output.
  • intersection, takes two sets and also produces a new one as output.
  • is_empty, takes a single set and returns true iff the set contains no items.
  • contains_item, takes a set and an item, and returns true iff the set contains the given item.

Additionally, as a data structure, we may include two other operations to mutate a given set:

  • add_item, takes a set and an item; after calling this function, the set contains the given item.
  • del_item, takes a set and an item; after calling this function, the set is guaranteed to not contain the item.

The wording of the last two operations was carefully chosen so that even if the item is/isn't contained in the set beforehand, it is clear what the result should be, so you can infer the behavior under all the cases.

Define the Set ADT

In this lab, we'll build sets containing strings. So you might have a set of artists: {"Matisse", "Degas", "Dubuffet", "Rodin"} or shoes: {"flip-flops", "sneakers", "slippers"}, etc.

Without beginning on a .cpp file, write a .h header file to define the operations that are useful for a set_of_str_t (cf. the stack example from the lecture).

A simple case

Write a bit of code, for example a short main function, that uses the properties you've defined. Here's a sample: 

#include "your_set_decl.h"

int main(int argc, char *argv[]) {
    set_of_str_t *fruity_set = create_set_of_str();
    add_item(fruity_set, "apple"); 
    add_item(fruity_set, "orange"); 
    add_item(fruity_set, "banana"); 
    add_item(fruity_set, "jackfruit"); 
    add_item(fruity_set, "jackfruit");  // add twice
    set_of_str_t *tasty_set = create_set_of_str();
    add_item(tasty_set, "jackfruit"); 
    add_item(tasty_set, "ice cream");  

    set_of_str_t *both_set = intersection(fruity_set, tasty_set);

    destroy_set_of_str(both_set);
    destroy_set_of_str(tasty_set);
    destroy_set_of_str(fruity_set);
}

Using the -c flag with g++, you should be able to compile this program, even though you can't link it—since there's no implementation of the set_of_str yet.

We probably want some sort of way to print out a set of strings, so that should be added to your interface.

Functionality to implement

Now write an implementation for set_of_str using a linked list. This may take a bit of time, because there are several functions that need to be implemented, but you can use a lot of code you wrote previously.

Remember: the crucial thing about a set is that it contains at most one instance of an element. So {"Torvalds", "Raymond"} ∪ {"Stallman", "Lessig", "Torvalds"} doesn't result with "Torvalds" ending up being stored twice. This is really the only challenge that needs to be overcome, the rest is similar to your previous work.

Extra functionality

A useful routine would be some way to iterate (or enumerate) through the set. It has already been suggested that a routine to print the set would be valuable. If you could iterate through the set, you'd be able to print off the items easily.

How could we do this in a general way? We'd like some sort of approach that wouldn't reveal (or leak) the implementation details. Remember the two versions of the stack: one using an array, another using a linked list? It would be interesting to give the user of your stack some way to "loop" through the items in the structure, without disclosing details about how you implemented the structure.

One approach is to declare another new type, something which hides the details of the underlying set. If that new type had functions to getFirst(), and to getNext(), and also to see if there are more, say via hasMore(), then you'd be able to write a for loop on that structure just by calling those functions.

The solution to this design problem is called an iterator, and we'll see more of them. But we're going to encounter them when dealing with objects rather than mere structs.
